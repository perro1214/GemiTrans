# GemiTrans 問題点・リスク一覧（デバッグメモ）

このドキュメントはコードレビューと静的解析に基づく **発見事項** の整理です。個別の不具合修正は、DEBUG 手順に従い **実行時ログで再現・検証** してから行うことを推奨します。

---

## 対応済み（コード反映済み）

| 項目 | 内容 |
|------|------|
| 1 | `translatePage` が `outcome`（`completed` / `skipped` / `cancelled` / `busy`）を返す。スキップ時は `TRANSLATION_SKIPPED`、完了時のみ `TRANSLATION_COMPLETE`。ポップアップは `TRANSLATION_SKIPPED` / `TRANSLATION_CANCELLED` を処理。 |
| 2 | バッチの一部失敗時は成功分のみ DOM 反映のうえ `throw` し、ポップアップに `TRANSLATION_ERROR` で伝える。 |
| 配置 | `manifest` 用の `content/content.js` をリポジトリ直下に配置（以前は誤配置フォルダのみに存在）。 |
| ポップアップ更新 | MV3 でコンテンツ発の `TRANSLATION_*` が SW にしか届かない場合に備え、`background/background.js` で `sender.tab` があるとき `chrome.runtime.sendMessage` により拡張ページへ転送。 |
| 3 | API エラー時に原文フォールバックせず null を返し、content.js で警告ログを出力するように変更（gemini.js, openrouter.js, lmstudio.js, content.js） |
| 4 | キャッシュキーの衝突耐性を向上させるため、ハッシュに原文の短いフィンガープリント（最初16文字+最後16文字）を追加（cache.js, background.js） |
| 5 | APIキーの平文保存について警告をポップアップUIに追加（popup.html, popup.css） |
| 6 | 自動翻訳のコスト・レート制限について警告をポップアップUIに追加（popup.html） |
| 10 | icons/*.png の実体を確認（icon16.png, icon48.png, icon128.png が存在することを確認） |

※ ポップアップを**閉じたまま**の進捗表示は従来どおり更新されない（別仕様）。

---

## 重大度: 高

### 1. 日本語ページを意図的にスキップした際に「翻訳完了」が通知される可能性

**状態:** 上記「対応済み」で修正済み。

**根拠（旧）:** `content/content.js` の `translatePage` は、翻訳先が日本語かつページ言語が日本語と判定された場合、**早期 `return`** する。このとき `state.isTranslating` は `true` になっていないが、メッセージハンドラ側では `translatePage(...).then(() => TRANSLATION_COMPLETE)` が成功扱いになり、**完了メッセージが送られる**。

**影響:** ポップアップで「翻訳が完了しました」と出るが、実際には翻訳していない（ユーザー混乱）。

**検証案:** 日本語サイトで翻訳先「日本語」を選び翻訳実行 → ポップアップの完了表示とページ内容が一致するか確認。

---

### 2. バッチの一部失敗時に部分的成功となり、エラーがユーザーに伝わりにくい

**状態:** 上記「対応済み」で、失敗時はエラー通知するよう変更済み。

**根拠（旧）:** `runWithConcurrency` の結果で `rejected` や `null` のバッチは `continue` でスキップされ、ログに `バッチN翻訳失敗` のみ。残りは適用されるため **ページは半分翻訳** のままになりうる。

**影響:** 静かな劣化（気づきにくい不整合）。

---

## 重大度: 中

### 3. API エラー時に原文へフォールバックし、失敗が目立たない

**根拠:** `utils/gemini.js` の `translateBatch` は、個別 `translateText` が失敗した場合 `catch` で **原文を返す**。ユーザーには「翻訳されなかった」ように見えるだけで、エラーと区別しづらい。

---

### 4. キャッシュキーの衝突リスク

**根拠:** `utils/cache.js` / `background/background.js` のメモリキーは 32bit 風ハッシュ＋短いプレフィックス。理論上 **異なる原文が同一キー** に衝突しうる（低確率だがデータ破損）。

---

### 5. API キーの保存形態

**根拠:** `chrome.storage.local` に API キーを平文で保存。拡張の権限モデル上は一般的だが、**端末上の平文ストレージ** である点はセキュリティ・コンプライアンス上の注意事項になりうる。

---

### 6. 自動翻訳のコスト・レート制限

**根拠:** `background/background.js` の `tabs.onUpdated`（`complete`）および `webNavigation.onHistoryStateUpdated` で、条件が揃うと **都度翻訳** が走る。タブを開き直す・SPA 遷移が多いサイトで **API 呼び出しが増えやすい**。

---

### 7. ポップアップを閉じている間の進捗・完了 UI

**根拠:** 進捗は `chrome.runtime.sendMessage` 経由でポップアップのリスナーが更新する前提。ポップアップが閉じていると **進捗バーは更新されない**（完了通知も同様）。仕様として許容できるが、期待とズレると「動いていない」ように見える。

---

## 重大度: 低 / メンテナンス

### 8. トークン使用量フィールドの API 仕様依存

**根拠:** `utils/gemini.js` は `usageMetadata.candidatesTokenCount` を出力トークンとして集計。Gemini API のレスポンス仕様が変わると **集計がずれる** 可能性がある。

---

### 9. 推定コスト表示の前提

**根拠:** `background/background.js` の `MODEL_PRICING` は固定テーブル。**実際の課金・プラン・リージョン** と一致しない場合がある（参考値としての限界）。

---

### 10. リポジトリ内に `icons/*.png` が見当たらない

**根拠:** `manifest.json` は `icons/icon16.png` 等を参照するが、ワークスペースのファイル一覧に PNG が無い。**パッケージング時にロード失敗** しうる（別ブランチで管理・ローカルのみ等の可能性あり）。配布前に要確認。

---
